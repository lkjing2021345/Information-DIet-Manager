# 大纲

## 整体概述

**在内容分类基础上，还必须引入情感分析、相似度分析和综合评估**；**这些模块分别解决了什么问题、为什么采用当前方案、如何保证稳定性与可解释性**；**系统输出如何支撑实际价值判断、目前有哪些局限、下一步如何改进**。

---

### 1. 总体概述
- 负责部分：重点负责情感分析模块、情感训练模块、文本相似度模块、综合评估模块
- 总体流程：浏览记录输入 → 内容分类 → 情感分析 → 相似度分析 → 综合评估 → 风险提示与报告输出
- 负责部分的功能定位：把原始浏览记录变成**可解释、可评分、可建议**的分析结果
- 这部分是系统从“识别内容”到“理解用户信息摄取结构”的主要结构
- 结果：支撑评估报告、风险识别与建议生成

---

## 2. 情感分析和分类模块采用以本地词典为主，模型为辅的设计

- `SentimentAnalyzer` 的核心设计是双路径：基于 `cntext` 的词典分析 + 基于自定义模型的预测（BERT）
- `ContentClassifier` 的核心设计同上：基于本地默认词典 + 自定义模型的预测（BERT）
- 采用词典原因：启动成本低、可解释性强、无需标注数据即可工作，适合作为默认主路径和兜底路径
- 模型的作用：在特定业务标注数据存在时，提高一致性，适配领域化表达
- 组合方式：先得到词典结果，再用模型预测对情感标签和置信度做修正

---

## 3. 情感分析模块实现了什么，如何保证一定有输出

- 统一输出结构：`SentimentScore` 负责词典打分，`SentimentPrediction` 负责统一预测结果
- 除情感标签外，还输出极性分数、正负词数量、置信度、匹配词列表、**细粒度情绪分布（未完成）**
- 提供多种分析：单条预测、批量预测、情绪分析、可读性分析、关键词提取、趋势分析和报告生成
- 稳定性设计：空文本判断、停用词缓存、自定义词典格式识别（YAML / CSV）、依赖缺失时保护性降级

---

## 4. 情感、分类模型训练模块单独设计

- 训练模块职责清晰：数据清洗、数据切分、标签映射、BERT 训练与验证、测试评估、模型产物保存
- 训练与推理解耦的意义：降低模块耦合，便于单独迭代模型，便于不同训练配置复用
- 训练过程中的工程设计：Hugging Face 固定缓存目录、模型自动缓存与复用、早停机制、学习率调度、梯度裁剪、可视化训练曲线保存
- 模型产物兼容性设计：`metadata.json`、`train_config.json`、`metadata.pkl`、`api_version` 版本检查
- 重点不是一次性训练脚本，而是形成可复用的训练流水线

### 可以有，麻烦就算了：
- 训练流程图：大模型并发蒸馏数据 → 数据清洗、标签重映 → 训练（添加早停机制防止过拟合）得到结果 → 微调参数观察macro_f1等指数变化 → 得到最佳参数

---

## 6. 相似度分析模块以 TF-IDF 为主的原因

- `SimilarityAnalyzer` 的核心能力包括：TF-IDF 余弦相似度、编辑距离相似度、可选词向量语义相似度、相似文本检索、重复检测、聚类分析、批量相似度计算
- 以 TF-IDF 为主的原因：当前任务输入多为中文短标题，关注的是重复、近似、同质化，而不是复杂深层语义推理
- TF-IDF 计算成本低，适合批量处理，稀疏表示可直接支撑重复检测和向量化计算
- 已预留更强语义方法接口：`use_word_vectors` 与平均词向量语义相似度

---

## 7. 综合评估器如何把多个底层分析结果变成“可解释的健康评估”

- `InformationQualityEvaluator` 负责把底层结果组织成健康等级、综合评分、风险警报、改进建议和导出报告
- 四维评估结构：多样性、情感健康、内容质量、时间分配
- 配置化设计：`EvaluatorConfig` 统一管理阈值和权重，支持更新配置与后续校准
- 风险识别不是只给结论，还包含风险类型、严重等级、关键证据、影响说明、改进建议
- 输出结构化报告：`EvaluationReport`，支持 JSON / Markdown / HTML 导出

---

## 8. 局限性和改进方向

- 局限1：评估器中的权重和阈值目前仍有规则设定成分，透明但还需要更多真实数据校准
- 局限2：词典法对反讽、隐喻、网络新词等处理有限，需要更多领域数据支持模型优化
- 局限3：TF-IDF 对**深层语义**近义改写识别有限，后续可引入句向量或更强语义模型
- 局限4：时间使用分析中有部分时长是基于记录数近似估计，后续可接入更真实的停留时长数据
- 后续改进方向：增加统一标注集和评测集、做更系统的参数校准、引入更强语义相似度模型、支持更细粒度个性化评估、加强前端可视化展示

### 可能的问题
1. 为什么情感分析模块采用“词典 + 可选模型”的组合，而不是统一使用深度学习模型？

   这两类方法各有**优势**。词典法启动成本低、可解释性强、不依赖标注数据，适合作为**基础能力和兜底方案**；模型法在有领域数据时能提升一致性和适配性。组合设计是为了在当前项目条件下兼顾**可用性和扩展性**。

2. `cntext` 在中文情感分析中的优势和局限分别是什么？

   优势是**对中文词典支持较成熟**，能快速输出正负词数、情绪类别，解释性较好。局限是对反讽、隐喻、上下文依赖较强的表达处理有限，**对新词和领域词的泛化能力也不如训练模型**。

3. 当词典结果与模型结果不一致时，你为什么采用当前这种置信度调整方式？

   不一致可以当作不确定性信号。如果两者冲突，就降低置信度；如果一致，就适度提高置信度。这样比直接覆盖更稳妥，也更符合多源信息融合的思路。

4. 你的情感训练模块目前如何保证训练产物与推理模块兼容？

   主要通过三点保证兼容：

   1. 保存**统一**的模型目录结构；
   2. 写出 `metadata.json` 和 `metadata.pkl`；
   3. 用 `api_version` 做**版本校验**。
      这样推理端在加载时可以识别训练产物格式，降低新旧版本不一致的问题。

5. 相似度分析为什么以 TF-IDF 为主？在什么情况下它会不如句向量方法？

   因为当前**输入以中文短标题**为主，任务更**偏向重复检测和同质化分析**，TF-IDF 在**速度、可解释性和实现复杂度上更均衡**。它的不足主要体现在深层语义表达上，比如同义改写、上下文语义接近但词面不同的情况，句向量通常会更有优势。

6. 你如何理解“相似度高”和“信息茧房”的区别与联系？

   相似度高是一个**局部文本特征**，说明内容之间接近；信息茧房是一个**更高层的结构性现象**，通常需要同时结合类别集中度、内容重复度和长期分布来判断。所以相似度高是信息茧房的证据之一，但不是充分条件。

7. 评估器中的阈值和权重来自哪里？它们是否具有主观性？

   目前主要来自规则化设定和项目场景下的**经验性初始化**，因此确实带有一定主观性。但它们被集中放在配置中，后续可以**基于真实数据和反馈继续校准**。

8. 时间分配分析中为什么有些时长是估算值？这会不会降低报告可信度？

   当前数据源主要是浏览记录，并不总是包含真实停留时长，因此用记录数和时间间隔做近似估计。它会影响精确度，但不影响系统先完成结构性分析。

9. 如果依赖库如 `cntext`、`torch`、`transformers` 缺失，系统如何确保仍然可运行？

   代码里做了**依赖检测和降级处理**。比如 BERT 相关依赖不可用时，模型路径会关闭，但词典路径仍可工作；输入为空或依赖缺失时也有统一默认返回和日志提示，尽量避免整个流程直接崩溃。

10. `EvaluationReport` 为什么要设计成 dataclass，并支持 JSON/Markdown/HTML 多格式导出？

    因为**评估结果本身是结构化数据**，用 dataclass 更方便统一字段和层级关系；多格式导出则是为了兼顾不同使用场景：JSON 适合程序调用，Markdown 适合文档汇报，HTML 适合展示和演示

11. 如果今后改成英文数据或跨语言数据，需要改动哪些部分？

    主要要改三类内容：

    1. 情感词典和分词工具；
    2. 训练模型和预训练语言模型；
    3. 类别规则和评估参数。
       整体框架可以保留，但底层语言资源需要替换。

12. 如果后续接入更强的大模型，你认为最值得先替换的是情感模块、相似度模块，还是评估器规则层？为什么？

    优先替换**相似度模块或情感模块的底层表示能力**更合适，因为这两部分直接受语义建模能力影响最大。评估器规则层目前更偏解释和组织逻辑，短期内不一定需要完全替换

13. 你认为你负责的模块中，当前最薄弱的一环是哪一部分？为什么？

    目前最薄弱的是**评估器中的参数校准**。因为底层分析已经有比较清晰的实现，但权重和阈值还缺少更系统的数据支撑。
